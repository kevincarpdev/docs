## Choose the right action

| Change type | Command |
| --- | --- |
| check inventory only | `bun run widgets:publish -- --list-only` |
| publish widget changes | `bun run widgets:publish` |
| publish one widget non-interactively | `bun packages/tooling/cli.ts widget publish --widgets <name> --yes --non-interactive` |
| validate widget contract | `bun packages/tooling/cli.ts widget validate <name>` |
| delete published widget or template | `bun packages/tooling/cli.ts widget delete ...` |
| clean up broken placements | `bun packages/tooling/cli.ts placement ...` |

## Primary publish flow

The recommended entrypoint is:

```bash
bun run widgets:publish
```

This flow can:

- detect whether package builds are stale
- run `bun run build` when needed
- publish selected widgets
- optionally run `bun run stencil:push`

## Common patterns

### List matched widgets without publishing

```bash
bun packages/tooling/cli.ts widget publish --list-only --storefront rustoleum-home
```

### Publish a specific widget

```bash
bun packages/tooling/cli.ts widget publish \
  --storefront rustoleum-home \
  --widgets categories \
  --yes
```

### Dry-run a publish

```bash
bun packages/tooling/cli.ts widget publish \
  --storefront rustoleum-home \
  --widgets categories \
  --dry-run \
  --yes
```

### Publish and update placements

```bash
bun packages/tooling/cli.ts widget publish \
  --storefront rustoleum-home \
  --widgets categories \
  --update-placements \
  --yes
```

### Publish and push the theme afterward

```bash
bun run widgets:publish -- --widgets categories --stencil-publish --yes
```

## How updates behave

### Widget template updates

By default, publish updates widget templates using the current template versioning mode.

You can control that behavior with:

- `--template-version-mode create-new-version`
- `--template-version-mode in-place`
- `--upgrade-template`

### Placement updates

Placement updates are **off by default**.

That means a normal publish does not automatically move or recreate placements. Use `--update-placements` when you want publish to apply:

- the widget's manifest `placements`
- explicit placement override flags such as `--template-file`, `--region`, `--entity-id`, and `--sort-order`

## When you also need `bun run stencil:push`

Widget publish alone is not enough when the change touches theme runtime code, including:

- React widget implementation changes
- theme wrapper changes
- registry changes
- Handlebars theme templates
- SCSS
- other theme assets

In those cases, publish the widget changes and then push the Stencil theme.

## Validation before publish

Run these before publishing meaningful widget changes:

```bash
# validate a single widget contract
bun packages/tooling/cli.ts widget validate <widget-name>

# or validate all widgets
bun packages/tooling/cli.ts widget validate

# build the workspace when package output changed
bun run build
```

## Required credentials

Publish and delete operations require:

- `BC_STORE_HASH`
- `BC_ACCESS_TOKEN`

`BC_CHANNEL_ID` is optional as a fallback channel, but you can override it per command with `--channel-id`.

## Safe mental model

- `widget.ts` changes affect the Page Builder contract
- `template.html` changes affect the widget handoff
- React and wrapper changes affect the storefront runtime
- publish updates BigCommerce widget resources
- `stencil:push` deploys theme code
