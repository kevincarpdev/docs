## Core files

Each package-backed widget folder should be read as a contract, not just a group of files.

| File | Role | Authored or generated |
| --- | --- | --- |
| `widget.ts` | source of truth for widget metadata, schema, translations, placements, and migrations | authored |
| `template.html` | Stencil SSR to React handoff | authored |
| `schema.json` | checked-in structural parity baseline | generated from `widget.ts` |
| `config.json` | compatibility artifact for publish and validation flows | generated from `widget.ts` |
| widget `.tsx` / `.styles.tsx` | runtime UI implementation | authored |
| widget `README.md` | widget-local implementation notes when needed | authored |

`meta.json` is not part of the current package-widget contract and should not be authored.

## `widget.ts` is the source of truth

The shared widget manifest utilities live in:

```text
packages/widgets/src/widgets/shared/widget-manifest.ts
```

The manifest exports the core authoring helpers:

- `defineWidget(...)`
- `translation(...)`
- `compose.anchorField(...)`
- `compose.button(...)`
- `compose.copy(...)`
- `compose.discountLabel(...)`
- `compose.icon(...)`

## What `widget.ts` controls

| Concern | Defined in `widget.ts` |
| --- | --- |
| widget name and description | yes |
| schema version | yes |
| Page Builder tabs, sections, and settings | yes |
| translation keys and default strings | yes |
| default placement targets | yes |
| schema migration hook | yes |

## Supported schema surface

The shared manifest layer supports these top-level schema constructs:

- `tab`
- `array`
- `hidden`

Important constraints:

- the schema must have an integer `schemaVersion >= 1`
- the current validation layer expects at most one array authoring surface
- tab and section labels must remain plain strings for editor compatibility
- array `entryLabel` values must also remain plain strings

## Example manifest pattern

```ts
import { compose, defineWidget, translation } from "../shared/widget-manifest.js";

export default defineWidget({
  description: "Example widget",
  name: "Example",
  schemaVersion: 1,
  placements: [{ channelId: 1, page: "pages/home", region: "", status: "active" }],
  schema: [
    {
      type: "tab",
      label: "Content",
      sections: [
        {
          label: "Content",
          settings: [
            {
              type: "input",
              id: "heading",
              label: translation("HeadingLabel", "Heading"),
              default: "",
            },
            ...compose.button(),
          ],
        },
      ],
    },
  ],
});
```

## `template.html` responsibilities

`template.html` should stay focused on the mount handoff:

- output one stable container element
- set `data-widget-type`
- serialize `data-widget-props`
- keep anchors in `data-widget-anchor` when needed

Do not move the full visual implementation into the template. The template exists to bridge BigCommerce widget rendering into the React runtime.

## Placements in the manifest

Placement objects support:

- `channelId`
- `entityId`
- `page`
- `region`
- `sortOrder`
- `status`

Those values are only applied during publish when placement updates are explicitly enabled.

## Generated artifacts

The tooling layer can load a widget manifest, emit artifacts, and write:

- `config.json`
- `schema.json`

That is why `widget.ts` should be treated as the authored source of truth.

## Translation behavior

`translation(...)` emits structured locale payloads into the generated schema translations surface.

Use it for user-facing labels that belong in the Page Builder editor contract.

Keep tab labels, section labels, and array entry labels as plain strings where the editor contract requires literal values.

## Shared compose helpers

The current shared compose helpers cover the most common contract fragments:

| Helper | Typical use |
| --- | --- |
| `compose.anchorField()` | optional in-page anchor |
| `compose.button()` | CTA text, variant, icons, and destinations |
| `compose.copy()` | heading, subheading, and body group |
| `compose.discountLabel()` | text/date/type discount label contract |
| `compose.icon()` | shared icon picker plus custom icon fallback |

Use them when they match the contract. Do not force a helper into a widget that needs a different authoring model.
