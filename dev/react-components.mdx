---
title: "Mapping Figma to MUI"
description: "How to translate Figma designs into repo-native MUI and shared UI code without copying raw CSS"
---

## Use the repo's component order

When a design needs to become code, choose components in this order:

1. an existing domain or widget component
2. an existing shared component from `packages/ui`
3. MUI composition with `styled()`
4. a new shared component in `packages/ui`
5. a widget-local or theme-local component only when the surface is not reusable

Do not start by copying raw CSS values from Figma.

## Token-first mapping

Theme tokens live in:

- `packages/ui/src/theme/theme.ts`
- `packages/ui/src/theme/tokens.ts`

Map design values through those surfaces first:

| Design intent | Repo surface |
| --- | --- |
| colors | `semanticColors` |
| typography | `typographyPresets`, `resolveTypographyPreset(...)`, `resolveTypographyScale(...)` |
| spacing rhythm | theme spacing and token spacing values |
| radius | `radius` |
| responsive breakpoints | `resolveViewportMinContentWidth(...)` and theme breakpoints |

## Typography mapping

Prefer repo typography presets over ad hoc font values.

Typical pattern:

```tsx
import { resolveTypographyPreset, resolveTypographyScale } from "@thesparklaboratory/rusto-ui/theme";
```

Practical rule:

- choose the closest preset first
- only override what the preset does not already express
- do not hardcode typography pixels when a preset or scale token already matches the intent

## Layout mapping

Break the Figma node tree into:

- section or surface container
- content-width container
- reusable child components
- slot regions

Then implement each layer intentionally in MUI instead of flattening the design into one giant style object.

## Icons

Use the shared icon surface:

```tsx
import { CaretRightIcon, resolveIcon } from "@thesparklaboratory/rusto-ui/Icon";
```

Rules:

- prefer Phosphor-backed shared icon exports
- use `resolveIcon()` only when the icon name is data-driven
- do not import icon packages directly in normal consumer code

## Widget-specific mapping

For widgets, split the design into two responsibilities:

- `widget.ts` for merchant-configurable inputs
- React component code for layout, states, token mapping, and behavior

Do not mirror every Figma layer as a widget setting. Only expose inputs that merchants should actually control.

## MUI implementation patterns

Prefer:

- named styled components
- package UI primitives
- semantic layout wrappers

Avoid:

- raw hex values copied from Figma
- direct pixel typography values when token helpers exist
- large one-off `sx` objects that become the only source of truth

## State checklist

Every serious design translation should account for the applicable states:

- default
- hover
- focus
- active
- selected
- disabled
- loading
- empty
- error
- permission-denied

If a state is not shown in the design file, make the behavior explicit during implementation instead of ignoring it.
