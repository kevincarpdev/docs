## Scaffold the widget

Use the repo CLI from the repo root:

```bash
bun packages/tooling/cli.ts widget create <widget-name>
```

Example:

```bash
bun packages/tooling/cli.ts widget create product-highlights
```

The scaffold creates:

```text
packages/widgets/src/widgets/<widget-name>/
├── widget.ts
├── template.html
├── <ComponentName>.tsx
└── <ComponentName>.styles.tsx
```

## What the scaffold does not do

The scaffold creates the folder and starter files only. You still need to:

1. implement the real widget UI
2. refine the schema in `widget.ts`
3. update `packages/widgets/src/index.ts`
4. update `packages/widgets/package.json` exports
5. add a registry entry in the theme
6. add a theme wrapper if the widget needs storefront data

## Recommended implementation order

### 1. Finalize `widget.ts`

Define:

- widget `name`
- `description`
- `schemaVersion`
- schema tabs, sections, settings, and translations
- optional `placements`

### 2. Build `template.html`

Keep the template focused on the handoff to React:

- set `data-widget-type`
- serialize `data-widget-props`
- keep `data-widget-anchor` separate when applicable

### 3. Implement the React widget

Use:

- shared UI from `packages/ui` first
- MUI composition second
- widget-local styled components only when the surface is widget-specific

### 4. Export the widget package entry

Add the new widget to:

- `packages/widgets/src/index.ts`
- `packages/widgets/package.json`

Without those exports, the theme registry cannot import the widget by package path.

### 5. Register the widget in the theme

Add an entry to:

```text
apps/storefronts/rustoleum-home/stencil-theme/assets/js/framework/widgets/registry.js
```

If the widget is presentational, the registry can load the package widget directly.

If the widget needs storefront data or theme runtime glue, add a wrapper under:

```text
apps/storefronts/rustoleum-home/stencil-theme/assets/js/components/widget-wrappers/
```

## Creation checklist

<Steps>
  <Step title="Scaffold">
    Run `bun packages/tooling/cli.ts widget create <widget-name>`.
  </Step>
  <Step title="Author the manifest">
    Replace starter schema fields in `widget.ts` with the real Page Builder contract.
  </Step>
  <Step title="Build the handoff">
    Update `template.html` so it emits the correct `data-widget-type` and serialized props.
  </Step>
  <Step title="Implement the UI">
    Build the React component with shared tokens and package UI primitives where possible.
  </Step>
  <Step title="Wire exports and registry">
    Update package exports and add the theme registry entry or wrapper.
  </Step>
  <Step title="Validate">
    Run widget validation, the touched-file Biome commands, and a workspace build if needed.
  </Step>
</Steps>

## Validation commands

```bash
# validate one widget
bun packages/tooling/cli.ts widget validate <widget-name>

# touched-file lint
bun run biome lint --fix --max-diagnostics=25 mintlify-docs/dev/creating-widgets.mdx

# touched-file format
bun run biome format --write mintlify-docs/dev/creating-widgets.mdx
```

## Practical guidance

- Keep `widget.ts` merchant-focused. Put visual structure in component code, not in schema sprawl.
- Prefer one top-level authoring array when repeatable content is necessary.
- Use shared compose helpers from the widget manifest utilities when they fit the contract.
- Do not create `meta.json` for package widgets.
- Treat `config.json` as generated compatibility output, not authored source.
