## Widget naming inconsistencies

### widget.yml vs widget.yaml

The TypeScript types (`src/types.ts`) define the meta file as `widget.yaml`, but the actual file created by `track.ts` and referenced by `sync-widgets.sh` is `widget.yml`. Always use `widget.yml` in practice.

### Widget type aliases

The widget registry in `widgets-loader.js` registers most widgets under multiple aliases:

```js
'herocarousel': () => import('./components/HeroCarousel.jsx'),
'herosectioncarousel': () => import('./components/HeroCarousel.jsx'),
'hero-carousel': () => import('./components/HeroCarousel.jsx'),
```

When creating a new widget, always register both the **PascalCase-lowered** and **hyphenated** versions of the type. The `data-widget-type` in `widget.html` determines which alias is used.

## Script inconsistencies

### publish-both-widgets.sh

Despite the name, `publish-both-widgets.sh` currently only publishes **HowItWorks**. The `README-WIDGETS.md` still describes it as publishing both Benefits and HowItWorks. Use `publish-all-widgets.sh` or `publish-widget.sh <name>` instead.

### stencil.conf.cjs

This file references `webpack.dev.js` and `webpack.prod.js` via `require()`, but these files may not be present in the widget-builder repo. This config is a Stencil hook that is only relevant when running inside the Stencil theme context.

## PDP rendering timing

### pdp-main-ready flag

Below-the-fold PDP widgets (HowItWorks, ProcessSection, etc.) wait for a `pdp-main-ready` CSS class before rendering. This class is added by the widget loader after `ProductView` finishes mounting.

If a widget is not appearing on a PDP, check that:
1. The widget's `data-widget-type` is registered in `widgets-loader.js`
2. The `ProductView` component mounted successfully (check browser console)
3. The widget is not stuck waiting for `pdp-main-ready`

### PDP widget priority

`product-image-gallery` widgets are prioritized first via `PDP_FIRST_WIDGET_TYPES`. All other PDP widgets load after the main product view is ready.

## Duplicate React prevention

Both `app.js` and `widgets-loader.js` need React. To prevent duplicate React instances (which cause hook errors), they share a single `getReactModules()` function exported from `widgets-loader.js`. Never import React directly in widget components — always use the shared modules.

## custom-widgets-backup is not deployed

The `custom-widgets-backup/` directory in the Stencil theme is listed in `.stencilignore`. It is purely a **version control backup** of widget source files. It is not included in the theme bundle that gets pushed to BigCommerce.

The actual widget templates live on BigCommerce's servers (uploaded via the Widgets API). The backup directory is for developer reference and git history.

## Unpublish command not wired up

The file `src/cli/deployment/widgetTemplateUnpublish.ts` exists and implements widget deletion, but it is **not registered** in `src/cli/index.ts`. To unpublish a widget, use the BigCommerce API directly.

## Image alt text normalization

The `?finish=` / `?color=` query parameter matching normalizes strings by lowercasing and stripping hyphens, underscores, and spaces. This means:

- `?finish=Natural` matches alt text `Natural`, `natural`, `NATURAL`
- `?finish=Light-Gray` matches alt text `Light Gray`, `lightgray`, `light-gray`

But it does **not** handle typos or abbreviations. The alt text must be an exact normalized match.

## Variant images get generic alt text

When a variant image is prepended to the gallery (because it does not already exist in the base image list), it receives a generic `alt: 'Product image'` rather than the variant name. This is a known limitation.

## MutationObserver for widget loading

`widgets-loader.js` uses a `MutationObserver` to detect new widget nodes added to the DOM (for SPA-like page transitions or lazy-loaded content). If you dynamically inject widget HTML after page load, the observer will pick it up automatically.
