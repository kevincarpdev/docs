---
title: "Developer gotchas"
description: "Current workflow pitfalls to avoid when working with widgets, placements, and theme deployment"
---

## `widget.ts` is authored, `config.json` and `schema.json` are not

Do not hand-edit generated widget artifacts as if they were the main source of truth.

- author `widget.ts`
- treat `schema.json` as the checked-in parity baseline
- treat `config.json` as generated compatibility output

## Placement updates are opt-in

Publishing a widget does not automatically reapply placements.

If you expect manifest placements to move or update live placements, you need `--update-placements`.

## Theme push and widget publish are separate

Publishing a widget does not deploy theme code.

If you changed:

- registry wiring
- wrapper code
- widget runtime code
- templates or SCSS

you still need `bun run stencil:push`.

## The publish git gate is strict

`bun run stencil:push` requires:

- branch `main`
- no local changes
- local `main` aligned with `origin/main`

It will not behave like a casual local preview command.

## Do not use `stencil init`

The repo expects config files to be placed directly in the storefront and theme directories.

Use the repo init command or place the files manually. Do not rely on the older interactive Stencil bootstrap workflow.

## Environment switching is multi-file

When you switch stores or environments, rotate the whole set:

- repo root `.env`
- storefront `.env`
- theme config
- theme secrets

Leaving one file pointed at the old environment creates hard-to-debug mismatches.

## Wrappers are for runtime data, not for everything

If a widget can render from authored props alone, load the package widget directly from the registry.

Add a theme wrapper only when storefront data or theme-side orchestration is actually needed.

## Keep the template thin

`template.html` should hand off props, not recreate the full UI system in Handlebars.

If a widget is getting visually complex in the template, move that work into the React implementation.
